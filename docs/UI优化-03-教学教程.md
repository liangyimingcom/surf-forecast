> 面向开发者：这些前端交互怎么实现、代码在哪、如何扩展、踩过的坑。全部在单文件 `web/浪报MVP.html` 内，纯前端附加式。

## 2. 设计原则

- **CSS 变量驱动主题**：主要面用 `var(--bg/--card/--ink*/--line)`，深色只需 `body.dark` 覆盖变量 + 少量白底类映射，避免逐处改色。
- **附加式、不进 render()**：`render()` 会整块覆写 `#strip/#hero/#story/#cards/#extras`，所有新交互挂在**独立容器 + 独立函数**，不触碰 render() 管辖 DOM。
- **localStorage 版本化 key**（`_v1` 后缀）：便于未来结构变更时平滑失效旧值。
- **脏值安全回退**：读 localStorage 后**白名单校验**（tab∈{live,report,other}、view∈{catalog,cams}），异常值回默认，杜绝坏状态卡死。
- **数据诚实**：hero 只显示引擎按浪点提供的字段（浪点名/日期/校准时间），删除写死的日出/水温/月相。

## 3. 如何新增一个 UI 交互（模式）

1. **grep 先确认**函数/元素是否已存在（防重复、防并发覆盖）。
2. HTML 加独立容器（若属某 tab，在 `TAB_OF` 注册其 id）。
3. CSS 用变量，深色在 `body.dark` 补覆盖。
4. JS 加渲染/切换函数 + 必要的 `localStorage` 持久化（版本化 key + 脏值回退）。
5. `web/e2e/new_features.mjs` 加断言（交互后 DOM/存储/可见性）。
6. `web/e2e/shots.mjs` 加截图项。
7. 跑 E2E 全绿 + pytest 不倒退。

## 4. 踩坑记录

- **并发双驱动写冲突**（本项目累计 3 次）：dashboard auto-nudge 与 `task_run` 同时驱动同一目标，并发写同一 tasks.md/HTML → strReplace 失败、出现自己没写的计划。**纪律：单一驱动器；dashboard 启动 loop 时绝不再调 task_run；每轮改前先 grep 实际文件为准，不信 tasks.md 自述。**
- **E2E 契约先行**：预写断言时须与实现用**同一 key/文案契约**（本轮 `sf_tab_v1`/「没有匹配」），否则实现方用不同名 → E2E 红。
- **子视图与 tab 联动**：进入 live tab 时 `showTab` 需重新 `applyLiveView()`，否则切回目录/直播状态丢失。
- **深色遗漏白底**：硬编码 `#fff` 的输入/按钮/弹层需在 `body.dark` 显式映射到 `var(--card)`，否则夜间刺眼。
