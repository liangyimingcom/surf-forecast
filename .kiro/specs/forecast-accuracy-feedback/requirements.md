# Requirements — Forecast Accuracy Feedback（昨日回看校验机制，v2 新增）

> EARS 格式。本 spec 定义"用用户真实体感校验预报准度"的信任机制。
> 已在 web/浪报MVP.html 以 `renderVerify()` + `rateYesterday()` 落地 MVP；本 spec 正式化并扩展到后端持久化与偏差校准。

## 文档目的

预报本身无法当场验证，但**昨天**用户能亲身验证。本机制把不可验证的未来，锚定在可验证的过去上：展示系统对昨日的预报 → 用户用体感自评 → 累积校准系统性偏差 → 让预报越用越准、越用越可信。

## 用户故事

- 作为**怀疑论者**，我想看到系统昨天预报了什么、对不对，再决定信不信它对未来的预报。
- 作为**老用户**，我希望我历次"预报 vs 体感"的反馈能校准出这个浪点的系统性偏差，让推荐更贴合实际。

## 1. 昨日历史浪报

**1.1** THE SYSTEM SHALL 用引擎历史模式（Open-Meteo `past_days=1`）取**昨日**(GMT+8 日界)的浪报，处理口径与预报完全一致。
**1.2** THE SYSTEM SHALL 展示昨日的综合分、五维评分、原始数据表、SVG 图表（与预报日同款，含离岸风编码）。
**1.3** THE SYSTEM SHALL 展示**系统当时对昨日的预报判断**（预期浪高/周期/风/最佳窗口/板型/综合结论），供用户对照记忆与体感。
**1.4** THE SYSTEM SHALL 确保"昨日"日期 = 今天(GMT+8) − 1，且不与预报区(今天起)重叠。

## 2. 用户自评

**2.1** THE SYSTEM SHALL 提供四档自评：**🎯准 / 📉偏乐观(实际没那么好) / 📈偏保守(实际比预报好) / 🤷没下水**。
**2.2** WHEN 用户选择某档，THE SYSTEM SHALL 给针对性反馈：
  - 准 → 模型标定可信，可放心参考本周预报；
  - 偏乐观 → 解释常见原因（近岸浅水打折、风/纯度体感更敏感），建议本周预期下调半档；
  - 偏保守 → 实际更好（局地涌浪聚焦/离岸梳面超预期），可上调半档但保留安全 margin；
  - 没下水 → 鼓励下次回来评，多积累以校准。
**2.3** WHERE 已登录，THE SYSTEM SHALL 持久化自评（用户、浪点、日期、档位、时间 GMT+8）。

## 3. 偏差校准（累积价值）

**3.1** WHEN 某用户/浪点累积 ≥N 条自评，THE SYSTEM SHALL 计算系统性偏差倾向（偏乐观/保守/准）。
**3.2** WHERE 存在明显偏差，THE SYSTEM SHALL 在后续预报中提示该浪点的体感修正建议（如"你常觉得此点偏乐观，建议下调半档看"）。
**3.3** THE SYSTEM SHALL 不自动篡改原始评分，仅以提示形式呈现校准建议（保持数据诚实）。

## 4. 呈现位置与时区

**4.1** THE SYSTEM SHALL 把昨日回看放在**预报内容之后**（不占据首屏黄金位）——校验是辅助信任的工具，不应挡在主信息前。
**4.2** THE SYSTEM SHALL 全程标注 GMT+8（昨日日期、自评时间）。

## 验收标准

| # | 标准 |
|---|------|
| F1 | 昨日日期 = 今天(GMT+8)−1，与预报区不重叠 |
| F2 | 昨日浪报口径与预报一致（同评分/图表/离岸编码） |
| F3 | 展示系统对昨日的预报判断供对照 |
| F4 | 四档自评 + 针对性反馈正确 |
| F5 | 登录用户自评持久化（含 GMT+8 时间） |
| F6 | ≥N 条后能给偏差校准提示，且不篡改原评分 |
| F7 | 昨日回看位于预报内容之后，非首屏 |
